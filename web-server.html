Proof of concept Web Server with an IMGUI

Introduction

Web technology is in my opinion one of the most annoying technologies to work with. Despite this I wrote a web server mostly from scratch to explore some ideas I had for how to program websites. The core idea I had was this:

What if you could write a whole web page just in plain C with an IMGUI without having to touch HTML, CSS and Javascript?

That would be very nice for me because C is my main progamming language and I love IMGUIs and hate HTML. Of course, HTML, CSS, and JS are required for the web server internals, but they would not be neccesary for the web programming part of the process in this setting. In this blog entry I will give a tour of the web server internals and highlight features I find interesting.

Firstly, all the source code can be found in THE SOURCE CODE WILL BE PUBLISHED SOON . The source code is in a unity build configuration, i.e. all source for each module is contained in a single compilation unit. To build the code invoke build.sh. The only supported operating system is Linux.

To begin with I will describe the modules system. The entry point of the server is located in main.c. On startup it loads the core module core.so(the core.csource file) and enters the main loop. The core.somodule then proceeds to setup sockets and configure the epoll interface. The advantage of putting that code in a separate module is that it allows for hot reloading, that is whenever core.sois compiled the server can, at runtime, load the new code without restarting. After having set up sockets the code then awaits connections from web browsers surfing to our site.

Upon succesfull connection with a client the server parses the HTTP and generates a response. The first part, parsing, is rather straightforward and I won't go into any further detail. The second part, generating a response, is where all the interesting stuff happens, and is what will be described in the following sections.

Memory Management

The fact that we can run out of memory and potentially crash the server is problematic. Thus we should not rely on uncommited virtual memory anywhere. Instead I commit all memory that is being used up front and additionally limit the memory used by individual connections. Upon reaching the memory limit, either the one in set by us in software or the hard one governed by the operating system, I longjump down the stack, free all the memory and (currently) refuse to respond to the client.

Currently the maximum total amount of memory used by the server in the case of many simultaneous connections is very high. The only action we take against this is limiting the maximum amount of simultaneous connections to 1024. We also don't take any action to prevent being OOM killed. A proper server implementation would of course require us to think about these things.

Immediate Mode User Interface

To begin with I will explain my process for the IMGUI. In a classical IMGUI the way you create a button widget by inserting something like the following snippet in your main loop: 

if(button("Press me")){
  do stuff
}

But this does not seem very straightforward in this setting with a web server. The way I made this work is by, on the server, generating a unique ID for each button and send this to the client in HTML (or some equivalent!, I will get into that in the next section!). The button upon being pressed on the client then sends a POST request (or something equivalent, I will get into this later as well) to the server containing the button ID. At that point the server reruns the IMGUI code and it is at this point that the call to buttonreturns truebecause the ID from the client and the one on the server matches. This solution allows us to get the normal behaviour we would expect from an IMGUI button. The output from that run of the IMGUI doesn't even need to be displayed to the client, it could be done entirely as something internal on the server and some other IMGUI code could be responsible for responding to the client. That gives some extra flexibility you would expect from an IMGUI.

UI Tree and Client-Server Communication

To retain focus on the right UI element we need to avoid calling appendChildfrom Javascript on elements and rows higher up in the hierarchy. To accomplish this the client retains the previous UI state and upon recieving new UI data from the server (in JSON format for simplicity) a diffing stage is performed. New elements are added in the proper places and old elements are removed. If any previously present elements change order we are essentially forced to perform an appendChildfor each element thus breaking focus. As that should be relatively uncommon I don't think this is a problem.

A different solution to this problem would be to manually keep track of what element is in focus. This seemed to be more complicated. For example retaining where the cursor is when taking input in a text box requires us to have full knowledge of the full state of the text box.

When the client wants to write back data to the server, e.g. because a button was pressed, it sends a message in a custom binary format back to the server.

All this communication is currently done over websockets. I originally used POST requests but using web sockets is much better because doing so doesn't need to reload the page whenever a button or something similar is interacted with, giving a nicer user experience.

Layouting

One of the core problems in user interface programming is doing layout in a satisfying manner. In our case we need to push layout through a pipeline starting with our UI interface and ending in HTML output. The way I decided to tackle this problem was to use the flexbox HTML layouting system and try to align our UI system interface to have this as the final output. The API design I use is heavily inspired by the Nuklear IMGUI library . We layout everything by prefacing every row by a call to 

void ui_layout_row(int n_columns, int flexes[/*n_columns*/]);
void ui_layout_row_center(int max_width_px);

specifying the proportion each element on the row will occupy or the maximum width in case of centered elements. I find that this approach works well for me.

Routing

The way I chose to do routing is very simple. I do it like the following code illustrates: 

if(path_consume(&path, "blog"){
  if(path.size == 0){
    serve_blog_entry_list();
  }else{
    String id = path;
    serve_blog_entry(id);
  }
}else if(path_consume(&path, "assets"){
  serve_asset(path);
}

In this code snippet path_consumereturns trueif the next part of the path matches the provided argument. E.g. if "blog/web_server"was passed then the first call to path_consumereturns trueand pathis set to "web_server".

This way of handling routing is really easy to combine with security. I did this in the example blog page (the page you are probably looking at at the moment, unless this is the static page version) for the admin page. The admin panel requires that you to give a specific key to authorise yourself. Extending the example above this amounts to something like the following: 

if(path_consume(&path, "blog"){
  if(path.size == 0){
    serve_blog_entry_list();
  }else if(path_consume("admin")){
    if(session.is_admin){
      serve_admin_panel();
    }else{
      serve_admin_authentication_page();
    }
  }else{
    String id = path;
    serve_blog_entry(id);
  }
}

Security and UI IDs

One huge security flaw with the standard way of doing IMGUIs is ID collisions. My approach to this problem is twofold. Firstly I simply detect whenever two widgets have exactly the same ID and never let those pages reach the web browser. This of course still allows for denial of service attacks by simply producing an ID collision. Secondly I use SHA-512 hashes to prevent ID collisions from arising in the first place. The reason this doesn't fully suffice is because of possible mistakes on the server, e.g. forgotting to push a unique id in a loop: 

for(i64 i = 0; i < count; i += 1){
  ui_text(UI_GEN_ID, ""); // This will generate many UI elements with the same ID!
}

The macro UI_GEN_IDgenerates an ID unique to the line of code in question. The solution here is of course to put something like ui_push_id_i64(UI_GEN_ID, i);and ui_pop_id()in the loop.

Logging

For logging I wanted trying out something more fun than simply dumping everything in a textual log file. Instead I chose to use a binary format (see the log.csource file) with a graphical viewer (source file log-viewer.cpp). The main reason for doing this was to try out something new as I have never seen anyone else do it (although it probably has been done before!). Additionally it was very easy to implement so no big costs there. Furthermore doing this allows for doing things like grouping together related log items, easily filtering output and allows for non-textual content like images.

Grouping things together is done by calling log_beginfollowed by logging what you want to log, and then ending with a matching call to log_end.

Rich Text Content

For rich text content I opted to use Markdown. Because I thought it would be easy to write a Markdown parser (surprise surprise it ended up being reasonably complicated) I did write one. It is not very technically interesting and is not particularly optimized (only some basic SIMD for the common case) so I won't bore anyone with any further details. Also, it only passes about half of the CommonMark tests and is incomplete in several areas. But it is good enough for my purposes. After parsing the Markdown document is converted into HTML which is then forwarded to the web browser viewing the website.

Because the default typesetting on the web sucks I implemented the Knuth-Plass line breaking algorithm on the server (or rather, reused some code I had written previously). What you are looking at at the moment is actually typeset using Knuth-Plass! The Knuth-Plass implementation from my earlier project ended up being very simple to reuse. The gist of the API is contained in the following code block: 

typedef struct KPBoxPos KPBoxPos;
struct KPBoxPos{
  f32 horz_pos;
  u32 line_number;
};

static bool kp_penalty(KPContext *kp, i32 penalty_value, f32 width_on_break); // Returns true if we break here
static KPBoxPos kp_box(KPContext *kp, f32 width);
static void kp_glue(KPContext *kp, f32 width, f32 stretch, f32 shrink);

If you have done any work with the Knuth-Plass algorithm you know what penalties, boxes and glue are. I won't explain those here. The approach here is to do two passes: One for the Knuth-Plass algorithm to get all the data it needs and one for the user of the API to get typesetting info back. On the first pass the procedures don't return any useful data but in the second pass they do return the final typesetting info. So in both passes you would invoke the penalty, box and glue routines in the exact same order but only save the result from the second round of invocations.

Limitations

One limitation of the server design is that it doesn't reload content instantly like IMGUIs tend to do. This would require either regular updates from the server, which I consider to be rather costly, or marking state as dirty, and then sending out new data to affected clients. Another limitation from the same system is that you can only have a single window or tab open on the same URL. For instance you might not be able to edit two blog posts at the same time. I have some strategies for solving this problem, like putting editing different blog posts on different URLs, but I never implemented this. As this kind of problem is present in most other websites too I don't think it is too problematic.

The server also has some technical limitations and problems. For example, we only allow 1024 simultaneous connections at the moment. We also only run a single thread and thus don't process requests as fast as we could. Also, a higher workload is put on the server because it runs the UI code. The memory limitations on connections can also be annoying, e.g. for uploading large images, because we don't do any streaming of data. We also don't have any good error messages for when the server runs out of memory.

Another set of limitations is how badly we do typesetting at the moment, the implementation both has bugs and contains hacks which don't work well for all cases. I do have solutions to most of the problems but haven't implemented them yet at this point in time.
